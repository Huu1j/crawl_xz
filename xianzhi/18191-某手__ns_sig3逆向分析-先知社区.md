### 去花

对于这个部分龙哥分析的很详细：<https://www.yuque.com/lilac-2hqvv/zfho3g/issny5?#yGLEd>

这里简要记录一下学习过程

可以看到jumpout，这种一般都是因为在so中添加了花指令，导致ida因为栈不平衡或者跳转地址是计算后的而无法正确反编译

![image.png](images/20250610142732-fd743ef4-45c3-1.png)

既然高级语言（伪代码）看不了，我们就来看一下汇编。可以看到这里不是正常函数开辟栈的操作。首先这个函数开辟的栈空间很小，其次arm64架构约定X0-X7 使用时不需要保存

![image.png](images/20250610142732-fd938c34-45c3-1.png)

仔细阅读后发现，

```
ADR X1, dword_462EC           =>X1=462EC
SUBS X1, X1, #0xC             =>X1=462E0
MOV X0, X1                    =>X0=462E0
ADDS X0, X0, #0x3C            =>X0=4631C
```

指令STR X0, [SP,#24]和LDP X2, X9, [SP,#16]实际上就是把X0的值传给X9，也就是说X9为4632C。所以最后br跳转到0x4632C

手动patch一下，把其他nop掉，然后BR X9改成B 0x4631C也就是直接跳转到真正的地址，之后函数就能正常反编译了

![image.png](images/20250610142732-fdc2b17e-45c3-1.png)

![image.png](images/20250610142733-fe317488-45c3-1.png)

接着编写脚本识别花指令，批量去花

```
import keystone
from keystone import *
import ida_bytes
import idaapi
import idc
import flare_emu


# 来自叶谷雨的代码 O(∩_∩)O
def binSearch(start, end, pattern):
    matches = []
    addr = start
    if end == 0:
        end = idc.BADADDR
    if end != idc.BADADDR:
        end = end + 1
    while True:
        addr = ida_bytes.bin_search(addr, end, bytes.fromhex(pattern), None, idaapi.BIN_SEARCH_FORWARD,
                                    idaapi.BIN_SEARCH_NOCASE)
        if addr == idc.BADADDR:
            break
        else:
            matches.append(addr)
            addr = addr + 1
    return matches


myEH = flare_emu.EmuHelper()
def getJumpAddress(addr):
    myEH.emulateRange(startAddr=addr, endAddr=addr + 36)
    return myEH.getRegVal("X9")


def makeInsn(addr):
    if idc.create_insn(addr) == 0:
        idc.del_items(addr, idc.DELIT_EXPAND)
        idc.create_insn(addr)
    idc.auto_wait()


def generate(code, addr):
    ks = Ks(keystone.KS_ARCH_ARM64, keystone.KS_MODE_LITTLE_ENDIAN)
    encoding, _ = ks.asm(code, addr)
    return encoding


matches = binSearch(0, 0, "E0 07 BE A9 E2 7B 01 A9")
for addr in matches:
    print("try:" + hex(addr))
    makeInsn(addr)
    targetAddr = getJumpAddress(addr)
    code = f"B {hex(targetAddr)}"
    bCode = generate(code, addr)
    nopCode = generate("nop", 0)
    ida_bytes.patch_bytes(addr, bytes(bCode))
    ida_bytes.patch_bytes(addr + 4, bytes(nopCode) * 9)
```

### 固定结果

接下来的思路是trace代码，通过结果反向追踪产生流程

通过执行unidbg代码，可以发现虽然我们传入了固定的参数，但是最终结果仍然是变化的。固定的结果才有利于我们结合trace和调试分析流程。所以首先是想办法将结果固定

猜测sig3生成过程中使用了时间戳或者随机数之类的值。so可以有很多方法获取时间戳和随机数,比如jni,库函数,系统调用(最常见)以及文件访问

```
gettimeofday
gettimeofday64
getrandom
```

在导入表中搜索一下，可以看到gettimeofday。因此接下来可以尝试修改一下unidbg源码中的gettimeofday来看看是否是时间戳对结果造成了影响

![image.png](images/20250610142735-ff73951a-45c3-1.png)

gettimeofday 是在unidbg-api\src\main\java\com\github\unidbg\unix\UnixSyscallHandler.java中实现的

由于so是64位的，修改64位的即可

def sub_word(word):
    """对 32 位字的每个字节使用 S-box 进行替换"""
    return (
        (sbox[(word >> 24) & 0xFF] << 24) |
        (sbox[(word >> 16) & 0xFF] << 16) |
        (sbox[(word >> 8) & 0xFF] << 8) |
        (sbox[word & 0xFF])
    )

def rot_word(word):
    """将 32 位字左循环移位8位"""
    return ((word << 8) & 0xFFFFFFFF) | (word >> 24)

def reverse_key_schedule(last_round_key_hex):
    """
    根据 AES-128 的最后一轮密钥逆推出原始密钥
    参数:
        last_round_key_hex: 32个十六进制字符的字符串，例如 "D014F9A8C9EE2589E13F0CC8B6630CA6"
    返回:
        原始密钥（32个十六进制字符）
    """
    # 初始化 44 个 32 位字
    words = [0] * 44

    # 将最后一轮密钥拆分成 4 个 32 位字，并放入 words[40..43]
    for i in range(4):
        words[40 + i] = int(last_round_key_hex[i*8:(i+1)*8], 16)

    # 反向推导，从 i=43 到 i=4，逐步计算出前面的字
    for i in range(43, 3, -1):
        if i % 4 == 0:
            # 当 i 是 4 的倍数，使用 SubWord(RotWord(...)) 和 Rcon
            temp = sub_word(rot_word(words[i-1])) ^ (rcon[i // 4 - 1] << 24)
            words[i-4] = words[i] ^ temp
        else:
            words[i-4] = words[i] ^ words[i-1]

    # 将前4个字组合成原始密钥
    original_key = ''.join(format(word, '08X') for word in words[0:4])
    return original_key

\*(int8x8\_t \*)&s[i] = vrev32\_s8(\*(int8x8\_t \*)((char \*)v255 + i \* 4));

问一下ai,

```
这条指令使用 ARM NEON SIMD 技术执行向量反转操作，具体功能是：
● 从源地址 (char *)v255 + i * 4 加载 8 个 8 位整数（组成一个向量）
● 对每个 32 位块内的 8 位元素进行位置反转
● 将结果存储到目标地址 &s[i]

1. 从地址 v255 + i*4 加载8个字节：[a0,a1,a2,a3,a4,a5,a6,a7]
2. 按32位块分组：[a0,a1,a2,a3] 和 [a4,a5,a6,a7]
3. 对每个块内的8位元素反转：
   - [a0,a1,a2,a3] → [a3,a2,a1,a0]
   - [a4,a5,a6,a7] → [a7,a6,a5,a4]
4. 将结果存储到地址 &s[i]
```

简单来说就是对a5重新排序后存储在s中了。但是值得注意的是s一次存了8字节，循环了8次，也就是a5实际上有64个字节, 那么a6（在刚才代码中取的位数一致）应该也是64个字节。
