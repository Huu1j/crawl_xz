## 1、HTTP 主机标头

HTTP Host 标头是自 HTTP/1.1 起强制的请求标头。它指定客户端想要访问的域名。例如，当用户访问 时https://baidu.com，其浏览器将编写包含 Host 标头的请求，如下所示：

```
GET /web-security HTTP/1.1
Host: baidu.com
```

在某些情况下，例如当请求已被中间系统转发时，Host 值可能会在到达预期的后端组件之前被更改。我们将在下面更详细地讨论这种情况。

## 2、HTTP Host 标头的用途

HTTP Host 标头的目的是帮助识别客户端想要与哪个后端组件进行通信。如果请求不包含 Host 标头，或者 Host 标头的格式存在缺陷，则会在传入路由到目标系统时存在问题。

### 虚拟主机

```
可能的情况是，单个 Web 服务器托管多个网站或应用程序。
这可能是由一个所有者管理的多个网站，但也有可能由不同所有者管理的网站托管在单个共享平台上。
这种情况比以前少见，但仍会出现在某些基于云的 Saas 解决方案中。
无论如何，尽管这些不同的网站都有不同的域名，但它们都与服务器共享一个IP 地址。
以这种方式托管在单个服务器上的网站称为“虚拟主机”。
对于访问网站的普通用户来说，虚拟主机通常与托管在自己的专用服务器上的网站没有区别。
```

### 通过中介路由流量

```
另一种常见情况是，网站托管在不同的后端服务器上，
但客户端和服务器之间的所有流量都通过中间系统路由。
这可能是一个简单的负载平衡器或某种反向代理服务器。
这种设置在客户端通过内容分发网络(CDN) 访问网站的情况下尤其普遍。
在这种情况下，即使网站托管在单独的后端服务器上，所有域名都会解析到中间组件的单个 IP 地址。
这带来了一些与虚拟托管相同的挑战，
因为反向代理或负载平衡器需要知道应该将每个请求路由到哪个适当的后端。
```

### HTTP Host 标头如何解决这个问题？

```
在这两种情况下，Host 标头都用于指定预期收件人。一个常见的类比是向住在公寓楼的人发送信件的过程。
整个大楼都有相同的街道地址，但在这个街道地址后面有许多不同的公寓，
每个公寓都需要以某种方式接收正确的邮件。
解决此问题的一个方法是简单地在地址中包含公寓号或收件人的姓名。
对于 HTTP 消息，Host标头具有类似的用途,
当浏览器发送请求时，目标 URL 将解析为特定服务器的 IP 地址。
当此服务器收到请求时，它会参考 Host 标头来确定目标后端并相应地转发请求。
```

## 3、HTTP host攻击

HTTP Host 标头攻击利用了易受攻击的网站，这些网站以不安全的方式处理 Host 标头的值。如果服务器隐式信任 Host 标头，但未能正确验证或转义，攻击者可能会使用此输入注入有害的有效负载，从而操纵服务器端行为。直接将有效负载注入 Host 标头的攻击通常称为“Host 标头注入"攻击。现成的 Web 应用程序通常不知道它们部署在哪个域上，除非在设置期间在配置文件中手动指定。当它们需要知道当前域时(例如，为了生成电子邮件中包含的绝对 URL)，它们可能会求助于从 Host 标头中检索域:

```
<a href="https://_SERVER['HOST']/support">Contact support</a>
```

标头值还可用于网站基础设施的不同系统之间的各种交互。由于 Host 标头实际上是用户可控制的，因此这种做法可能会导致许多问题。如果输入未正确转义或验证，则Host 标头可能会成为利用一系列其他漏洞的潜在载体，最明显的是:

```
Web 缓存中毒
特定功能中的业务逻辑缺陷
基于路由的 SSRF
经典的服务器端漏洞，例如 SQL注入
```

## 4、HTTP Host 头漏洞是如何产生的？

HTTP Host 标头漏洞常因用户误信 host 不可修改，导致验证不足或转义问题，攻击者可轻易修改；即使处理更安全，Host 也可能被其他注入标头覆盖，且网站可能忽视默认支持的标头；许多漏洞源于基础设施配置不安全，因集成第三方技术不了解配置选项及影响。

## 5、如何使用 HTTP Host 标头测试漏洞

探测 Host 头注入漏洞时，第一步是测试提供任意不可识别域名的反应，但部分拦截代理会从 Host 标头取目标 IP，导致改 Host 即发往其他 IP、测试受阻；Burp Suite 可分离 Host 标头与目标 IP，支持自定义 Host 头，确保请求抵达预期目标。

```
目标 URL 显示在面板顶部(用于 Burp Repeater 和 Proxy 拦截)或 Burp Intruder 中的“目标”选项卡上。
您可以通过单击铅笔图标手动编辑目标。
```

提供意外 Host 标头时，有时仍能访问目标网站（因服务器有默认 / 后备配置，若目标为默认网站可研究其操作及利用性）；但篡改 Host 标头常导致无法访问（前端服务器或负载均衡器不知转发方向，出现 “Invalid Host header” 错误，通过 CDN 访问时更可能），此时应尝试下文技术。

### 检查验证缺陷

请求可能被阻止而非收到 “Invalid Host header” 响应（如网站验证 Host 标头与 TLS 的 SNI 匹配），但未必安全；应了解网站解析 Host 标头的方式，可能发现绕过漏洞（如解析时省略端口，只验证域名，可通过非数字端口注入有效负载）。

```
GET /example HTTP/1.1
Host: vulnerable-website.com:bad-stuff-here
```

其他网站会尝试应用匹配逻辑来允许任意子域名。在这种情况下，您可以通过注册以与白名单相同的字符序列结尾的任意域名来完全绕过验证：

```
GET /example HTTP/1.1
Host: notvulnerable-website.com
```

或者，您可以利用已经入侵的不太安全的子域名：

```
GET /example HTTP/1.1
Host: hacked-subdomain.vulnerable-website.com
```

有关常见域验证缺陷的更多示例，请查看有关[规避常见 SSRF 防御](https://portswigger.net/web-security/ssrf#circumventing-common-ssrf-defenses)和[Origin 标头解析错误](https://portswigger.net/web-security/cors#errors-parsing-origin-headers)的内容。

### 发送模糊请求

验证与利用主机的代码可能在不同组件或服务器上，利用其检索主机标头的差异，可发出模糊请求，不同系统看到的主机不同。

以下只是如何创建模糊请求的几个例子。

#### 注入重复的主机头

尝试添加重复 Host 标头通常会被阻止，但因浏览器少发此类请求，可能有未预料情况；不同系统处理方式不同，常一个标头覆盖另一个，系统意见不一致时可能产生可利用差异。

```
GET /example HTTP/1.1
Host: vulnerable-website.com
Host: bad-stuff-here 修改的host
```

假设前端优先考虑标头的第一个实例，但后端优先考虑最后一个实例。在这种情况下，可以使用第一个标头来确保您的请求被路由到预期目标，并使用第二个标头将您的有效负载传递到服务器端代码。

#### 提供绝对 URL

尽管请求行通常指定所请求域上的相对路径，但许多服务器也配置为理解绝对 URL 的请求。

同时提供绝对 URL 和 Host 标头所造成的歧义也会导致不同系统之间的差异。正式而言，在路由请求时应优先考虑请求行，但实际上并非总是如此。可以像利用重复的 Host 标头一样利用这些差异。

```
GET https://vulnerable-website.com/ HTTP/1.1
Host: bad-stuff-here
```

请注意，可以尝试不同的协议。服务器有时会根据请求行包含 HTTP 还是 HTTPS URL 而做出不同的行为。

#### 添加换行

通过空格缩进 HTTP 标头可能发现奇怪行为，部分服务器将其视为前一标头值的一部分，其他则忽略；处理不一致常导致不同系统间出现差异。

```
GET /example HTTP/1.1
    Host: bad-stuff-here
Host: vulnerable-website.com
```

网站可能阻止多个 Host 标头请求，但可缩进其中一个标头绕过；前端若忽略缩进标头，请求视为普通的[vulnerable-website.com](https://vulnerable-website.com/)，后端若忽略前导空格且优先第一个标头，这种差异可通过 “包装的” Host 标头传递任意值。

### 注入主机覆盖标头

即使无法用模糊请求覆盖 Host 标头，也可通过其他专用于此的 HTTP 标头注入有效负载；网站常经中介系统访问，后端收到的 Host 标头可能是中介域名；前端可能注入 X-Forwarded-Host 标头含原始 Host 值，许多框架会优先引用，即使无前端使用也可能如此；可利用 X-Forwarded-Host 注入恶意输入，绕过对 Host 标头的验证。

```
GET /example HTTP/1.1
Host: vulnerable-website.com
X-Forwarded-Host: bad-stuff-here
```

尽管X-Forwarded-Host这是此行为的事实标准，但您可能会遇到具有类似用途的其他标头，包括：

```
X-Host
X-Forwarded-Server
X-HTTP-Host-Override
Forwarded
提示:
从安全角度来看，需要注意的是，有些网站（甚至可能是您自己的网站）无意中支持这种行为。
这通常是因为他们使用的某些第三方技术默认启用了一个或多个此类标头。
```
