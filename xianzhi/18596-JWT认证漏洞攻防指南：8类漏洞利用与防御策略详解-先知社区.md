```
该实验室使用基于JWT的机制来处理会话。由于实现缺陷，服务器不会验证它收到的任何JWT的签名。
```

#### ​Lab2: JWT authentication bypass via flawed signature verification

##### 2.1、漏洞描述

```
该实验室使用基于JWT的机制来处理会话。服务器的配置不安全，无法接受未签名的JWT。
要解决实验室问题，请修改您的会话令牌以访问/admin的管理面板，然后删除用户carlos。
您可以使用以下凭据登录自己的帐户：wiener:peter
```

##### 2.2、漏洞复现

​

在 Burp 中，转到Proxy > HTTP history选项卡并查看登录后`GET /my-account`请求。观察您的会话 cookie 是 JWT。

![image.png](images/20250808153723-8606dee2-742a-1.png)

在 Burp Repeater 中，将路径更改为`/admin`并发送请求。观察发现，只有以`administrator`用户身份登录时才能访问管理面板。

修改sub=administrator，response为401

![image.png](images/20250808153724-869fa1f4-742a-1.png)

选中 JWT 的 header，然后使用 Inspector 将`alg`参数的值改为`none`。点击Apply changes

修改alg=none，response为401

![image.png](images/20250808153725-87364c1a-742a-1.png)

删除签名，并保持后面.符号。执行修改后的jwt token，在response中发现/admin/delete?username=carlos接口

![image.png](images/20250808153726-87b54fba-742a-1.png)

#### Lab6、[JWT authentication bypass via kid header path traversal](https://portswigger.net/web-security/jwt/lab-jwt-authentication-bypass-via-kid-header-path-traversal)

##### 6.1、漏洞描述

```
该实验室使用基于JWT的机制来处理会话。为了验证签名，服务器使用JWT标头中的kid参数从其文件系统中获取相关密钥。
要解决这个问题，请创建一个JWT，让您可以访问/admin的管理面板，然后删除用户carlos。
您可以使用以下凭据登录自己的帐户：wiener:peter
```

##### 6.2、漏洞复现

访问admin，返回401

![image.png](images/20250808153739-8fc83212-742a-1.png)

转到Burp 主选项卡栏中的 JWT 编辑器密钥选项卡。

单击“新建对称密钥”。

在对话框中，单击“生成”以生成 JWK 格式的新密钥。请注意，您无需选择密钥大小，因为这稍后会自动更新。

将属性的生成值替换`k`为 Base64 编码的空字节 ( `AA==`)。请注意，这只是一种解决方法，因为 JWT 编辑器扩展不允许您使用空字符串对令牌进行签名。

![image.png](images/20250808153740-901ec29e-742a-1.png)

返回`GET /admin`Burp Repeater 中的请求并切换到扩展生成的JSON Web Token消息编辑器选项卡。

在JWT的header中，将参数的值改为`kid`指向`/dev/null`文件的路径遍历序列：

`../../../../../../../dev/null`

在 JWT 有效负载中，将声明的值更改`sub`为`administrator`。

在选项卡的底部，单击“签名”，然后选择您在上一节中生成的对称密钥。

确保选中了“不修改标头”选项，然后单击“确定”。修改后的令牌现在使用空字节作为密钥进行签名。

```
//header
{
    "kid": "../../../../../../dev/null",
    "alg": "HS256"
}

//body
{
    "iss": "portswigger",
    "exp": 1727838676,
    "sub": "administrator"
}
```

![image.png](images/20250808153740-9064391e-742a-1.png)

修改后，访问admin界面成功

![image.png](images/20250808153741-90b045ac-742a-1.png)

完成Lab6

#### Lab7、[JWT authentication bypass via algorithm confusion](https://portswigger.net/web-security/jwt/algorithm-confusion/lab-jwt-authentication-bypass-via-algorithm-confusion)

##### 7.1、漏洞描述

```
该实验室使用基于JWT的机制来处理会话。它使用一个强大的RSA密钥对来签名和验证令牌。然而，由于实现缺陷，这种机制容易受到算法混淆攻击。
要解决这个问题，首先需要获取服务器的公钥。这是通过标准端点公开的。使用此密钥对修改后的会话令牌进行签名，该令牌使您可以访问/admin的管理面板，然后删除用户carlos。
您可以使用以下凭据登录自己的帐户：wiener:peter
```

##### 7.2、漏洞复现

访问admin，返回401

![image.png](images/20250808153742-9143020c-742a-1.png)

转到标准端点`/jwks.json`并观察服务器公开包含单个公钥的 JWK Set

![image.png](images/20250808153743-91b2e7ca-742a-1.png)

在 Burp 中，转到Burp 主选项卡栏中的 JWT 编辑器密钥选项卡。

单击“新建 RSA 密钥”。

在对话框中，确保选择了JWK选项，然后粘贴刚刚复制的 JWK。单击“确定”保存密钥。

![image.png](images/20250808153743-91f6b1c6-742a-1.png)

右键单击刚刚创建的密钥条目，然后选择“将公钥复制为 PEM”。

使用解码器选项卡对此 PEM 密钥进行 Base64 编码，然后复制生成的字符串

![image.png](images/20250808153743-921c691e-742a-1.png)

返回Burp 主选项卡栏中的 JWT Editor Keys选项卡。

单击“新建对称密钥”。在对话框中，单击“生成”以生成 JWK 格式的新密钥。请注意，您无需选择密钥大小，因为这稍后会自动更新。

将 k 属性的生成值替换为您刚刚创建的 Base64 编码的 PEM。

![image.png](images/20250808153743-923ac8c0-742a-1.png)

返回`GET /admin`Burp Repeater 中的请求并切换到扩展生成的JSON Web Token选项卡。

在 JWT 的标头中，将参数的值更改`alg`为`HS256`。

在有效载荷中，将声明的值更改`sub`为`administrator`。

在选项卡的底部，单击“签名”，然后选择您在上一节中生成的对称密钥。

确保选中了“不修改标头”选项，然后单击“确定”。修改后的令牌现在使用服务器的公钥作为密钥进行签名。

```
//header
{
    "kid": "c97a0686-7923-412a-b88d-ceb133737bb8",
    "alg": "HS256"
}

//body
{
    "iss": "portswigger",
    "exp": 1727840053,
    "sub": "administrator"
}
```

![image.png](images/20250808153744-9277381e-742a-1.png)
