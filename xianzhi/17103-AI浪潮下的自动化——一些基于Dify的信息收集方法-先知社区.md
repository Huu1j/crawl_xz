# AI浪潮下的自动化——一些基于Dify的信息收集方法-先知社区

> **来源**: https://xz.aliyun.com/news/17103  
> **文章ID**: 17103

---

Dify是一个开源的的生成式AI应用创新引擎，其中最吸引我的是他的工作流。实际攻防演练信息收集过程中，我们会反复操练形成我们自己的工作流，完全可以基于大模型和工作流自动化。所以基于此开发了一些Dify信息收集Workflow，方便流程自动化，目前基于v0.15.3实现了三个基础DSL。

![image.png](images/07fa48ca-6399-3264-8018-60ef2c878045)

## 0x01 资产测绘引擎**数据聚合**

![image.png](images/05bc2587-e75a-3e9d-bdd3-090a1e1b03fc)

网络空间安全引擎种域名、ip的聚合。目前主要支持了Hunter、Quake、零零信安。

注意事项：

> 1. 需要配置 Hunter、Quake、零零信安的 API Key；
> 2. 目前分页数据的无法全部读取。

## 0x02 **团队协作**

![snake.png](images/7c0acd73-3dbc-3f56-93f5-7564f5ed73e8)

针对在某些活动场景下大数据量的信息收集，以飞书文档（飞书团队版支持50人，足够了）为依托展开的信息收集。实现流程如下：

1. 在dify输入项目名称和目标列表；
2. 创建项目总目录；

1. 常见目标文档
2. 递归爬取目标的ip的域名；

3. 飞书群聊中发送消息。

![snake-p9.png](images/07f53164-88bd-3d66-abd8-035983afdeaf)

注意事项：

> 1. 创建飞书应用需要提供，同时群通知机器人需要提供 hook地址 ；
> 2. 依赖于 0x01 的资产测绘引擎，所以需要配置 Hunter、Quake、零零信安的 API Key；
> 3. 目前还不支持对子公司展开文档创建。

## **0x04 Dify workflow使用小技巧**

1. 工作流中HTTP访问尽量使用“代码执行”，而不是封装的“HTTP请求”，部分json格式的处理比较头疼；

![image.png](images/da5086bc-b341-36da-9a0c-89e2cee54861)

2. Dify v1.0.0 版本之后插件需要联网下载的，建议使用 v0.15.3；

3. Dify的知识库不如RAGflow，如果考虑使用知识库，建议针对Ragflow做流程开发。
